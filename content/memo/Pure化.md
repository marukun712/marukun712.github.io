## 1. 背景と課題：Web 2.0の限界と分散型SNSのジレンマ

### 1.1 Web 2.0プラットフォームがもたらした構造的課題
現在のWeb 2.0型中央集権プラットフォームは、以下の3つの致命的な課題を抱えています。
* **データの独占と収益構造の歪み:** ユーザーが生成したコンテンツや活動データをプラットフォームが囲い込み、それを原資として広告等で不当に高い利益を得る構造。
* **API制限によるエコシステムの破壊:** 外部開発者が構築したサードパーティエコシステムを、プラットフォーム側の都合（マネタイズや囲い込み）で一方的に遮断・破壊する。
* **BANによる実データ・ソーシャルグラフの喪失:** アカウントの一方的な凍結により、ユーザーが長年蓄積してきたアイデンティティ、実データ、および人間関係のネットワーク（ソーシャルグラフ）が瞬時に喪失するリスク。

### 1.2 分散型SNSの限界と「コールドスタート問題」
MastodonやNostr、Blueskyに代表される分散型SNSは、データの主権をユーザーに取り戻すことで上記の課題を解決しようと試みました。しかし、依然として以下の構造的な限界を突破できていません。
* **ネットワーク効果の壁:** 圧倒的なネットワーク効果と濃密なソーシャルグラフは、依然としてX（旧Twitter）などの既存プラットフォームに残留しています。
* **コールドスタート問題:** プラットフォームに依存しない新しい分散型スペースを作っても、誰もいない（あるいは認知の伝播が極めて遅い）ため、コミュニティが自律的に成長しにくい。

既存プラットフォームの持つ「ネットワーク効果・ソーシャルグラフ」を維持・利用しつつ、**「実データの保護機構（あるいはデータ抽出・表現の権利）」だけを分散的に担保する方法はないか？** という問いが、本書の出発点です。

## 2. コアコンセプト：「Pure（ピュア）化」の定義

### 2.1 「Pure（ピュア）」とは何か
プラットフォームが提供するサービスから、UI、広告、タイムラインの操作アルゴリズム、プロプライエタリな表示レイヤーなど、**「プラットフォーム側の都合で付加されたノイズ」をすべて削ぎ落とし、データが本来持つ「純粋な意図（Intent）」のみを抽出した状態**を「Pure」と定義します。

### 2.2 JSONと自然言語：意図を表現するプリミティブ
* **JSONというPure:** かつてのAPIが重宝された理由は、それがUIや装飾から完全に独立し、データの構造と意図のみを返す「Pureな存在」だったからです。
* **自然言語というPure:** 人間が交わす自然言語は、意図のみを表す最もプリミティブかつ普遍的な言語です。現在のテクノロジーにおいて、自然言語は人間だけでなく機械（AI/LLM）にとっても「可読な意図の表現形態」となりました。

## 3. 意図の抽出メカニズムと「決定論的理解」

### 3.1 LLMのジレンマと「決定論的言語理解」への回帰
自然言語から意図を抽出する際、大規模言語モデル（LLM）への過度な依存は新たな「リソース競争」と「中央集権化」を招きます。富める者がより強力なモデルを独占する構造は、Web 2.0の構造と変わりません。

したがって、WebのPure化においては、生成AIのような確率論的・巨大なモデルではなく、**「局所的かつ決定論的な言語理解アルゴリズム」**を採用すべきです。
* **Readabilityアルゴリズムの派生:** Webページの雑音（ナビゲーション、広告、フッター）を排除して本文だけを抽出する「Readability」の手法をセマンティック（意味論的）に拡張する。
* **軽量・局所的なパース:** 巨大なサーバーを必要とせず、クライアントサイドや安価なエッジサーバーで動作する、ルールベースおよび軽量な自然言語処理（NLP）の組み合わせ。

### 3.2 アナログホール問題（Analog Hole）の応用
プラットフォームは、どれほどデータを囲い込み、APIを制限しようとも、**最終的にはユーザー（人間）に情報を「見せる」必要**があります。
* プラットフォームは意図（コンテンツ）の周囲にいくらでも広告やノイズを配置できますが、**「意図そのもの」を除外してはサービスとして成り立ちません。**
* ユーザーの画面にレンダリングされる以上、そこには必ず「意図」が露出しています。これを機械的・決定論的に逆コンパイルしてPureなデータへと還元することを、プラットフォーム側が完全に阻害することは不可能です（特にオープンなWeb標準においては）。

## 4. テクニカル・アプローチ：アクセシビリティ・ツリーの活用

WebのPure化を最も現実的、かつ社会正義に基づいて実現するための鍵が**「アクセシビリティ・ツリー（Accessibility Tree）」**です。

```
[ 既存プラットフォームのHTML / DOM ]
            │ (ノイズ、広告、UI装飾の混入)
            ▼
[ アクセシビリティ・ツリー ]  ◄── 社会通念上・法的に排除不可能なレイヤー
            │ (スクリーンリーダー等のための「純粋な構造と意図」)
            ▼
[ Pure化されたデータ（JSON / 構造化自然言語） ]
```

### 4.1 なぜアクセシビリティ・ツリーなのか
1.  **社会通念および法的強制力:** 障がい者やスクリーンリーダー（音声読み上げ）のためのアクセシビリティ（WAI-ARIA等）は、現代のWebにおいて「なくてはならない」社会通念であり、多くの国で法的に義務付けられています。プラットフォーム側がこれを意図的に破壊・スクランブル化することは、社会的・ブランド的に極めて困難です。
2.  **セマンティクスの結晶:** アクセシビリティ・ツリーは、DOMから見た目の装飾を削ぎ落とし、「ここにボタンがある」「ここに投稿本文（テキスト）がある」という**純粋な構造と意図のみをブラウザが解釈したツリー構造**です。

### 4.2 実装の展望
プラットフォームのアプリ版はプロプライエタリなバイナリであるためPure化は困難ですが、Web版（Desktop/Mobile Web）はブラウザの制御下にあります。拡張機能（Extension）やヘッドレスブラウザを介してアクセシビリティ・ツリーにアクセスし、そこから「投稿内容」「投稿者」「タイムスタンプ」というPureな意図を決定論的に抽出すれば、API制限を完全に無効化してデータを分散側に引き戻すことができます。

## 5. ユースケース：分散SNSのコールドスタート問題の解消とPOSSEの拡張

### 5.1 POSSE（Publish on Own Site, Syndicate Elsewhere）の双方向的拡張
インディWeb（IndieWeb）の基本原則である**POSSE**は、「自分のサイトに書き、他所（中央集権プラットフォーム）に配信する」という一方向の思想でした。

「Pure化」は、これを**双方向（Bidirectional POSSE）**へと拡張します。
* **吸い上げ（Ingest）:** 中央集権プラットフォーム（X等）のタイムラインをアクセシビリティ・ツリー経由でPure化し、自身の分散型スペース（Nostrや独自プロトコル）へとシームレスに同期・マージする。
* **ネットワーク効果のハッキング:** ユーザーは分散技術で作られた自由なUI/UXを使いながらも、裏側ではXにいる人々の「意図」を読み、それに対して自身の分散スペースから応答を差し戻す（あるいはPOSSEで同期する）。

これにより、分散コミュニティが「プラットフォームの中に引き籠もって分散技術について議論する」という矛盾した現状を打破できます。プラットフォームのネットワーク効果にタダ乗りしながら、依存度だけをゼロに近づけることが可能になります。

## 6. 結論：これからのWebの相互運用性のための新設計原則

Webの健全な相互運用性（Interoperability）を保つために、私たちはプラットフォームに対して「APIを公開しろ」と懇願するのをやめるべきです。代わりに、新たな設計原則を突きつける必要があります。

> **「APIを公開せよ、ではない。意図を意図として表現・還元できるエンドポイント（あるいは構造）を必ず残せ」**

もしプラットフォームがそれを拒み、意図を隠蔽したりアクセシビリティを軽視したりするならば、**「誰かが勝手にそれをPure化し、オープンなエンドポイントとして再公開するだけ」**です。

### 思想的スタンス
この試みは、世界全体を強制的にひっくり返そうとする革命ではありません。
**「中央集権の恐怖やリスクに気づいた人たち」「分散技術の本質的な楽しさを知っている人たち」が、プラットフォームの暴政に怯えることなく、それでもなお既存の世界（マジョリティのネットワーク効果）と豊かに関わり続けるための、静かなる「脱出用トンネル」の構築**なのです。
